name: Validate

# Render is connected directly to this GitHub repo and auto-deploys on push
# to main - there is no deploy step here. This workflow is just a cheap
# sanity check (valid JSON/HTML, no obviously broken file paths) that runs
# before Render's own deploy kicks in, so problems show up in the PR/commit
# checks instead of only on the live site.

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Check package.json is valid JSON
        run: node -e "JSON.parse(require('fs').readFileSync('package.json','utf8')); console.log('package.json OK')"

      - name: Check render.yaml exists
        run: test -f render.yaml && echo "render.yaml OK"

      - name: Check core entry points exist
        run: |
          test -f index.html
          test -f src/core/app.js
          test -f src/core/config.js
          echo "Entry points OK"

##---FILE-START---## README.md
# NEXUS [FPS]

A fast, fluid arena FPS for the browser. Inspired by **Flick FPS**, **Rivals**,
and **Krunker** — high-speed movement (slide-dash, jump, slide-jump), one-shot
sniper duels, and a stylized low-poly look (not literal 8-bit pixels — a
clean, technical, HUD-driven art direction).

## Current milestone

**Landing → Loading → Menu**, AI bots only. Multiplayer is intentionally
disabled (`FEATURE_FLAGS.multiplayer = false` in `src/core/config.js`) until
the engine and netcode are ready.

- `index.html` — entry point
- `src/core/app.js` — screen flow controller (landing → loading → menu → game)
- `src/ui/landing/` — 3D title screen (Three.js, low-poly rotating map)
- `src/ui/loading/` — bridge/progress screen
- `src/ui/menu/` — main menu (Play / Shop [locked] / Settings)
- `src/core/game.js` — placeholder game mount point (next milestone)

## Why no build step

This project is edited in **Spck Editor** and deployed by pushing straight
to GitHub, with **Render** serving the repo as a static site. So there is
**no bundler, no `npm install` step, no transpilation** — everything is
plain HTML/CSS/JS, and Three.js is loaded from a CDN via an import map in
`index.html`. Keep it that way: don't introduce a build tool without
updating the deploy workflow and Render's start command to match.

## Local preview

Any static file server works, e.g.:

```bash
npx serve .
# or
python3 -m http.server 8080
```

Then open the printed URL. ES module imports require serving over
`http://`, not opening `index.html` directly via `file://`.

## Folder structure

```
public/          static assets (models, textures, audio, fonts, icons...)
src/core/        app bootstrapping, global config, the (future) game loop
src/ui/          every screen: landing, loading, menu, in-match HUD, overlays
src/player/      first-person controller, movement, health, input
src/weapons/     weapon definitions, starting with the sniper
src/engine/      physics, collision, raycasting, effects, audio, particles
src/ai/          AI bot behavior (current focus, ahead of multiplayer)
src/maps/        map loading, spawn points, the AI-bot training map
src/economy/     coins, rewards, skins (UI exists, locked until live)
src/multiplayer/ client-side networking (inactive — flagged off)
src/save/        local profile + settings persistence
src/utils/       shared helpers
server/          future backend (auth, database, matchmaking, websocket) —
                 not active yet; the game currently runs entirely client-side
tests/           placeholder for future unit tests
docs/            changelog, controls, roadmap, weapon design notes
```

## Design direction

See `src/ui/landing/landing.css` header comment for the full token system
(color palette, type pairing, and the "signature element" — the PLAY button
doubles as a scope-charge control, snap-zooming into the map on click to
mirror the sniper's fast-scope).

See `docs/roadmap.md`.

# Controls

## Movement
- **Sprint**: toggle (no walking — sprint is the default ground-movement state)
- **Slide**: crouch while sprinting
- **Slide-jump**: jump while sliding → higher jump than a normal jump (tuned
  subtle, not an exaggerated bunny-hop — see `MOVEMENT.slideJumpBonus` in
  `src/core/config.js`)
- **Crouch**: standalone button, also the slide trigger when sprinting

## Combat
- **Scope**: fast scope-in, detailed sight
- **Fire**: tap-to-fire by default; "hold to scope, release to fire" is an
  alternate style available in Settings

## Touch HUD (mobile)
- Virtual joystick for movement
- Buttons: fire, scope, jump, crouch
- All toggleable in Settings → HUD (see `src/ui/menu/menu.js`)

## Settings currently exposed
- Scope style (detailed / minimal)
- Shoot button behavior (tap / hold-to-scope)
- Joystick on/off
- Crouch button on/off
- Jump button on/off

Advanced settings (graphics, audio mixing, key rebinding) are locked for
now — see `FEATURE_FLAGS.settingsAdvanced` in `src/core/config.js`.

# Weapons

## Sniper (current focus)

The only weapon implemented for the AI-bot test milestone. See exact values
in `src/core/config.js` → `WEAPONS.sniper`.

**Feel**: fast scope-in (not a slow zoom-creep), detailed sight with visible
range marks/reticle. One shot kills.

**Tracer**: not instant-hitscan visually — a fast tracer arcs upward and
curves back down before landing. The *landing point* is always exactly
where the scope was aimed at the moment of firing (no real bullet drop or
spread is applied to the actual hit registration — the arc is a visual
flourish layered on top of an instant-resolve hit check).

Implementation split:
- `src/weapons/scope.js` — scope-in timing + FOV change
- `src/weapons/tracer.js` — the curved visual projectile
- `src/weapons/hitDetection.js` — the actual (instant) hit resolution
- `src/weapons/sniper.js` — ties the three together into one weapon

## Planned weapons

Not yet started. Will follow the `src/weapons/weaponBase.js` interface once
the sniper is feature-complete and feels right.

// core/app.js
// Entry point. Boots the landing screen first. Everything else (loading,
// menu, game) is lazy-loaded only when needed, so the initial page stays
// light. This file owns ONLY screen transitions - no game logic lives here.

import { mountLanding } from "../ui/landing/landing.js";

const screens = {
  landing: document.getElementById("landing-root"),
  loading: document.getElementById("loading-root"),
  menu: document.getElementById("menu-root"),
  game: document.getElementById("game-root"),
};

function showOnly(name) {
  for (const key of Object.keys(screens)) {
    screens[key].hidden = key !== name;
  }
}

async function goToLoading() {
  showOnly("loading");
  const { mountLoading } = await import("../ui/loading/loading.js");
  mountLoading(screens.loading, {
    onComplete: goToMenu,
  });
}

async function goToMenu() {
  showOnly("menu");
  const { mountMenu } = await import("../ui/menu/menu.js");
  mountMenu(screens.menu, {
    onPlay: goToGame,
  });
}

async function goToGame() {
  showOnly("game");
  const { mountGame } = await import("./game.js");
  mountGame(screens.game);
}

// Boot sequence
showOnly("landing");
mountLanding(screens.landing, {
  onPlay: goToLoading,
});

};

  walkSpeed: 0,             // no walking - sprint is the default ground state (toggle)
  sprintSpeed: 9.5,         // units/sec, tuned fast on purpose (Krunker/Flick-like)
  slideSpeed: 13,           // initial slide burst speed
  slideDuration: 0.45,      // seconds before slide friction takes over
  slideFriction: 14,        // deceleration applied during slide
  jumpVelocity: 6.2,        // base jump impulse
  slideJumpBonus: 1.35,     // multiplier applied to jump velocity if jumping out of a slide
  airControl: 0.65,         // fraction of normal control while airborne
  gravity: 18,
  crouchHeight: 0.9,        // fraction of standing height
  crouchSpeedMultiplier: 0.55,
  sprintIsToggle: true,     // per design brief: sprint toggles on/off, no hold-to-walk
};
